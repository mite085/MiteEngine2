## 1. 项目背景

### 1.1 项目起源

用户之前花费约半年时间基于 OpenGL、GLFW、ImGui 开发了一个三维渲染引擎，项目代号"MiteEngine"。该引擎在初期采用了基础的 ECS 架构和渲染管线，能够实现基本的渲染功能。然而，随着项目推进，在其基础上搭建材质编辑、骨骼动画等高级功能时遇到了显著困难，主要原因在于初期架构设计过于简陋，各模块之间耦合严重，缺乏清晰的分层和抽象，导致扩展性不足。

### 1.2 当前状态

项目目前处于重新开始的初期起步阶段。用户希望汲取之前的经验教训，在新一轮开发中采用更合理的架构设计，以支持更高级的功能开发和长期演进。

### 1.3 团队构成

项目开发采用人类与 AI Agent 协作的模式。Claude Opus 负责顶层设计和架构规划，具体实现由 AI Agent 团队执行，人类负责架构决策和代码审查。

**关键背景**：

- 用户之前使用 Claude Opus 的 AI Agent 进行开发，代码质量远超个人水平
- 用户的新角色：参与架构设计和迭代决策，不参与细节实现
- AI Agent 的代码需要经过对抗式代码审查以确保质量
- **重要**：架构设计将委托给更强大的模型执行，需求文档不应包含具体架构设计内容

### 1.4 开发时间约束

**当前状态**：用户目前已就业，项目时间从全职开发转变为兼职开发。

**时间分配**：

- 用户可投入时间：每天工作之余参与架构设计讨论、迭代决策
- 细节实现：全权委托给 AI Agent 团队执行
- 代码审查：人类负责最终质量把控

**影响**：由于是兼职开发，项目里程碑需要适当宽松，避免因时间压力导致架构质量妥协。

## 6. 相位渗透玩法需求分析

### 6.1 玩法定义

**已确认的理解**：玩家可以在多个"空间相位"之间切换，切换后进入另一个"相位空间"，部分物体会改变可见性/碰撞属性。

**参考机制**：《Biphase》的红黑世界切换

### 6.2 相位数量要求

**已确认**：至少需要支持 3 个相位：Normal（正常）、PhaseA、PhaseB

- Normal：正常状态
- PhaseA：相位A状态
- PhaseB：相位B状态
- PhaseA 和 PhaseB 之间可以相互切换
- 架构需要支持未来扩展到更多相位

### 6.3 功能需求

**核心机制**：

| 机制 | 描述 | 对引擎的需求 |
|------|------|--------------|
| 相位切换 | 玩家可以在多个空间相位之间切换 | 多状态渲染、物理系统扩展 |
| 渗透移动 | 在特定相位下可以穿过常规障碍物 | 碰撞系统扩展、状态感知 |
| 环境交互 | 某些元素仅在特定相位可见或可交互 | 物体可见性控制、状态绑定 |
| 谜题机制 | 利用相位切换解开环境谜题 | 关卡编辑器、状态触发器 |

**渲染系统需求**：

- 多状态渲染：每个物体可能支持多个相位状态，不同状态有不同的材质/效果
- 相位视觉效果：相位切换过渡效果（溶解/扭曲/淡入淡出）
- 动态可见性：基于相位状态的物体显示/隐藏

**物理系统需求**：

- 相位相关碰撞：物体在特定相位下不可见/不可碰撞
- 跨相位交互：某些物体可以与多个相位交互
- 可穿越区域：特定相位下可穿越的障碍物

**实体系统需求**：

- 相位组件：定义实体的相位属性
- 碰撞相位组件：定义实体的碰撞属性
- 交互组件：定义可交互元素的相位属性

### 6.4 参考游戏案例

| 游戏案例 | 相关机制 | 可借鉴点 |
|----------|----------|----------|
| 《使命召唤》Phase Shift | 维度切换躲避伤害 | 相位持续时间、冷却机制 |
| 《Biphase》（双相） | 红黑世界颜色切换 | 视觉风格区分、状态切换反馈 |
| 《Upside-Down Dimensions》 | 多维度切换解谜 | 维度间关系设计、谜题结构 |

## 7. 编辑器功能需求分析

### 7.1 编辑器定位

**双角色定位**：

| 角色 | 需求 | 设计策略 |
|------|------|----------|
| 开发者（用户） | 高效制作游戏内容 | 功能完整、快捷键支持 |
| 最终用户 | 友好体验 | 简化界面、预设模板 |

### 7.2 核心设计原则

| 原则 | 描述 |
|------|------|
| 相位优先 | 所有编辑操作需要考虑相位维度 |
| 实时反馈 | 修改立即可见 |
| 数据驱动 | 一切皆数据 |
| 可扩展 | 便于添加新功能 |

### 7.3 核心功能需求

#### 7.3.1 场景编辑器

- 物体管理（创建、删除、复制、变换）
- 物体层级与分组
- 相位感知编辑（过滤、批量设置、预览）

#### 7.3.2 相位编辑器

- 相位状态配置（可见性、碰撞、交互、效果）
- 相位切换触发器（区域触发、交互触发、条件触发）
- 相位预览（单相、叠加、动画）

#### 7.3.3 关卡流程编辑器

- 关卡列表管理
- 区域/房间管理
- 检查点与目标

#### 7.3.4 SDF编辑器

**已确认**：SDF编辑器是必须的，用于可视化创建SDF几何体。

- 雕刻工具（形状笔刷、修饰工具、擦除工具）
- 布尔运算（并集、差集、交集、平滑融合）
- 相位SDF功能

**SDF在引擎中的定位**：

| 维度 | SDF渲染能力 | SDF编辑器 |
|------|-------------|-----------|
| 位置 | 引擎核心 | 编辑器工具 |
| 必须性 | **玩法必须** | **关卡编辑必须** |
| 说明 | 实现相位穿透效果 | 可视化创建关卡 |

### 7.4 独创性功能需求

#### 7.4.1 AI指令生成系统

**已确认边界**：AI生成的是"配置数据"，非最终3D模型资源。

- 场景生成（自然语言描述关卡）
- 资产生成（自然语言描述资产）
- 逻辑生成（自然语言描述逻辑）

#### 7.4.2 节点式逻辑系统

- 事件节点、动作节点、条件节点
- 相位专用节点

#### 7.4.3 热重载系统

**已确认**：这是开发时的效率工具，不是运行时功能。

- Shader热重载：开发者修改Shader文件 → 编辑器检测到变更 → 重新编译并应用到当前场景
- 场景热重载
- 脚本热重载
- 材质热重载

### 7.5 功能优先级

**P0（核心基础）**：

- 场景编辑器：物体创建/变换/删除
- 场景编辑器：相位属性配置
- 相位编辑器：相位切换触发器
- 相位预览：实时相位预览
- 热重载：Shader热重载
- **SDF渲染能力**：实现相位穿透效果

**P1（核心功能）**：

- 关卡流程：区域/房间管理
- **SDF编辑器**：可视化创建关卡
- 热重载：场景/材质热重载
- AI生成：模板系统

**P2-P3**：布尔运算、节点系统、高级功能
