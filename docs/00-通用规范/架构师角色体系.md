# 架构师角色体系

**版本**：v1.0

**用途**：定义架构设计阶段的多层架构师角色体系

---

## 角色矩阵

| 角色 | 数量 | 职责 | 输出 |
|------|------|------|------|
| **总架构师** | 1人 | 整体架构规划、模块划分、技术选型 | 总体架构文档 |
| **接口人** | 1人 | 模块间接口协调、解耦审查、整体一致性 | 接口规范文档 |
| **模块架构师** | N人 | 各模块的详细架构设计 | 模块架构文档 |

---

## 角色详解

### 1. 总架构师 (Chief Architect)

```
职责：
- 制定项目的整体技术方向
- 定义模块划分方案
- 做出关键技术选型决策
- 制定编码规范和设计原则
- 审核各模块架构设计

输出：
- 总体架构文档
- 模块划分方案
- 技术选型报告
- 设计原则和约束

视角：全局、顶层、长期
```

### 2. 接口人 (Integration Lead)

```
职责：
- 负责模块与模块之间的接口设计
- 确保模块间解耦
- 审查模块间的依赖关系
- 协调模块架构师之间的沟通
- 确保整体功能完备性
- 解决模块边界问题

输出：
- 模块接口规范
- 依赖关系图
- 集成测试方案

视角：模块边界、集成点、数据流

关键能力：
- 理解每个模块的职责和边界
- 识别模块间的数据流和依赖
- 发现并解决耦合问题
- 确保接口的稳定性和可扩展性
```

### 3. 模块架构师 (Module Architect)

```
职责：
- 负责单个模块的详细架构设计
- 定义模块内部的子系统和组件
- 设计模块的接口（供其他模块使用）
- 选择模块内部的技术方案
- 与接口人协作确保模块边界清晰

输出：
- 模块架构文档
- 模块内部设计
- 模块接口定义

视角：模块内部、局部、详细

示例模块（以渲染引擎为例）：
- 渲染模块架构师
- 物理模块架构师
- ECS模块架构师
- 编辑器模块架构师
- 资源管理模块架构师
```

## 协作流程

```
总架构师
    ↓
制定模块划分方案
    ↓
┌─────────────────────────────────────────────┐
│ 模块架构师（各模块并行）                      │
│   渲染模块架构师 → 渲染架构文档              │
│   物理模块架构师 → 物理架构文档              │
│   ECS模块架构师 → ECS架构文档               │
└─────────────────────────────────────────────┘
    ↓
接口人
    ↓
审查模块间接口
协调依赖关系
确保解耦和功能完备
    ↓
总架构师
    ↓
审核整体架构
确认发布
```

## 接口人的核心职责

### 1. 模块解耦

```
解耦目标：
- 模块间无直接引用（通过接口/事件通信）
- 模块可独立编译和测试
- 模块可替换实现

解耦方法：
- 使用接口抽象
- 使用事件/消息系统
- 使用依赖注入
- 避免全局状态
```

### 2. 接口设计

```
接口设计原则：
- 最小化接口（只暴露必要功能）
- 稳定性优先（接口变更影响大）
- 自文档化（接口名清晰表达意图）
- 前后兼容（考虑扩展性）
```

### 3. 依赖管理

```
依赖规则：
- 禁止循环依赖
- 依赖方向必须清晰
- 跨模块调用必须通过接口
- 依赖层级不超过2层
```

## 决策权限

| 决策项 | 总架构师 | 接口人 | 模块架构师 |
|--------|---------|--------|-----------|
| 模块划分 | ✅ 决定 | 建议 | 参与 |
| 模块接口 | 审核 | ✅ 决定 | 建议 |
| 模块内部设计 | 审核 | 了解 | ✅ 决定 |
| 技术选型 | ✅ 决定 | 建议 | 参与 |
| 模块间依赖 | 审核 | ✅ 决定 | 建议 |
| 编码规范 | ✅ 决定 | 执行 | 遵守 |

## 输出物清单

### 总架构师输出

```
1. 总体架构文档
   - 项目技术方向
   - 模块划分方案
   - 技术选型决策
   - 设计原则

2. 架构评审报告
   - 各模块架构审核
   - 技术风险评估
```

### 接口人输出

```
1. 模块接口规范
   - 所有模块的公开接口
   - 接口使用示例
   - 版本兼容性说明

2. 依赖关系图
   - 模块依赖图
   - 数据流图
   - 集成点清单

3. 集成测试方案
   - 集成测试用例
   - 边界条件测试
```

### 模块架构师输出

```
每个模块的架构文档：
- 模块职责定义
- 内部子系统划分
- 核心数据结构
- 关键算法设计
- 对外接口定义
- 性能考量
```

## 协调者职责

在架构设计阶段，协调者需要：

```
1. 启动总架构师
   - 分配任务：制定模块划分方案

2. 启动模块架构师（并行）
   - 各模块独立设计

3. 启动接口人
   - 协调模块间接口
   - 审查解耦情况

4. 收集所有架构文档
   - 汇总审查

5. 启动总架构师审核
   - 最终审核

6. 决策记录
   - 更新迭代文档
```

## 角色协作示例

### 场景：设计渲染引擎架构

```
阶段1：总架构师
- 划分模块：渲染、物理、ECS、资源、编辑器
- 技术选型：SDF、Raymarching、EnTT

阶段2：模块架构师（并行）
- 渲染架构师：设计渲染管线、光照系统
- 物理架构师：设计碰撞检测、相位系统
- ECS架构师：设计实体管理、组件系统

阶段3：接口人
- 审查渲染模块和物理模块的接口
- 确保ECS可以驱动渲染和物理
- 设计事件系统用于模块通信

阶段4：总架构师审核
- 确认整体架构合理
- 批准发布
```

## 经验反馈点

```
架构阶段常见问题：
1. 模块划分不合理 → 模块间耦合严重
2. 接口定义不清晰 → 集成时大量修改
3. 依赖关系混乱 → 循环依赖或深层依赖
4. 遗漏边界情况 → 模块交互时崩溃
5. 过度设计 → 接口过于复杂

这些都通过接口人的审查来预防
```

---

**此角色体系确保架构设计既有全局视角，又有模块深度，同时保证模块间解耦**
