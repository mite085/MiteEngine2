# Agent 权限管理

**版本**：v1.0

**用途**：定义 Agent 的系统权限，一次性授权，后续无需人类干涉

---

## 1. 权限原则

### 一次性大授权

```
人类一次性授予协调者和项目经理系统权限：
- 后续所有授权操作无需人类确认
- 权限范围明明确定
- 超出权限范围需人类审批

### 破坏性操作确认原则（强制）

```
以下操作属于破坏性操作，执行前必须向人类确认：

1. git reset --hard（会删除未提交的修改）
2. git reset --soft HEAD~1（会回退提交）
3. git stash drop（会删除暂存的修改）
4. 删除非空目录
5. 删除 git 未跟踪的文件（未 add 的新文件）
6. 其他可能造成不可逆后果的操作

确认话术：
"即将执行 [操作]，这会 [后果]。是否确认执行？"
```

### 破坏性操作检查清单

```
协调者在执行任何操作前，必须自问：

□ 这个操作会删除任何未提交的修改吗？
□ 这个操作会回退任何已提交的代码吗？
□ 这个操作会永久删除任何文件吗？
□ 这个操作的后果可以撤销吗？

如果有任何一项为"是"，必须向人类确认后再执行。
```

### 权限等级

| 等级 | Agent | 权限范围 |
|------|-------|---------|
| L1 | 协调者 | 所有权限 |
| L2 | 项目经理 | 大部分权限 |
| L3 | 其他 Agent | 无系统权限，仅执行任务 |

## 2. 协调者权限（L1）

### 文件操作

```
✅ 权限内：
- 创建/删除/修改任何文件
- 创建/删除目录
- 移动/重命名文件

❌ 权限外：
- 无
```

### Git 操作

```
✅ 权限内：
- git clone（拉取分支）
- git checkout（切换分支）
- git add / git commit（提交到本地）
- git merge / git rebase（合并分支）
- git branch（分支管理）

❌ 权限外：
- git push（推送到远程）
- git pull（从远程拉取）
```

### 特殊规则：本地提交原则

```
规则：
1. 所有 commit 只提交到本地仓库（git commit）
2. 不执行 git push（推送到远程）
3. 不执行 git pull（从远程拉取）
4. 推送操作由人类手动执行

原因：
- 避免频繁打扰人类确认
- 保持本地提交灵活性
- 人类可自行决定推送时机
```

### 构建操作

```
✅ 权限内：
- cmake 配置和生成
- make / ninja 构建
- 运行可执行文件
- 安装依赖

❌ 权限外：
- 无
```

### 环境配置

```
✅ 权限内：
- 安装/更新系统包
- 配置环境变量
- 安装开发工具
- 配置构建工具链

❌ 权限外：
- 无
```

### 网络操作

```
✅ 权限内：
- 拉取第三方库（git clone、wget、curl）
- 下载依赖包
- API 调用

❌ 权限外：
- 无
```

## 3. 项目经理权限（L2）

### 文件操作

```
✅ 权限内：
- 创建/删除/修改项目文件
- 创建/删除目录

❌ 权限外：
- 删除项目根目录
- 删除 .git 目录
```

### Git 操作

```
✅ 权限内：
- git add / git commit
- git push（需协调者审核后）
- git pull

❌ 权限外：
- git merge / git rebase
- 删除分支
- 强制推送（force push）
```

### 构建操作

```
✅ 权限内：
- cmake 配置和生成
- make / ninja 构建
- 运行测试

❌ 权限外：
- 修改系统配置
- 安装系统级依赖
```

### 环境配置

```
✅ 权限内：
- 配置项目级环境变量
- 安装项目依赖

❌ 权限外：
- 修改系统级配置
- 安装系统包
```

## 4. 权限矩阵

| 操作 | 协调者 | 项目经理 | 其他 Agent |
|------|--------|---------|-----------|
| 创建文件 | ✅ | ✅ | ❌ |
| 删除文件 | ✅ | ✅ | ❌ |
| 修改文件 | ✅ | ✅ | ❌ |
| git commit | ✅ | ✅ | ❌ |
| git push | ⏸️ 手动 | ⏸️ 手动 | ❌ |
| git merge | ✅ | ❌ | ❌ |
| git force push | ❌ | ❌ | ❌ |
| git reset --hard | ⚠️ 确认 | ❌ | ❌ |
| git reset --soft | ⚠️ 确认 | ❌ | ❌ |
| git stash drop | ⚠️ 确认 | ❌ | ❌ |
| cmake 构建 | ✅ | ✅ | ❌ |
| 安装依赖 | ✅ | ✅ | ❌ |
| 安装系统包 | ✅ | ❌ | ❌ |
| 修改系统配置 | ✅ | ❌ | ❌ |
| 创建分支 | ✅ | ✅ | ❌ |
| 删除分支 | ✅ | ❌ | ❌ |

注：
- ⏸️ 手动 = 由人类手动执行
- ⚠️ 确认 = 执行前必须向人类确认

## 5. 权限外操作处理

### 处理流程

```
Agent 遇到需要超出权限的操作：
    ↓
检查权限矩阵
    ↓
    ├─→ 权限内 → 直接执行
    │
    └─→ 权限外 → 报告协调者
            ↓
        协调者判断
            ↓
        ├─→ 可授权 → 执行并记录
        │
        └─→ 需人类审批 → 向人类请求审批
```

### 权限外示例

```
需要人类审批的操作：
- 删除整个项目目录
- 修改 .git/config 中的远程仓库
- 推送敏感信息到远程
- 安装可能影响系统的包
- 其他可能造成不可逆后果的操作
```

## 6. 审计日志

### 记录所有敏感操作

```
每次执行敏感操作后，记录：
- 操作时间
- 操作者
- 操作内容
- 操作结果
```

### 日志格式

```markdown
## 审计日志

### 2026-03-09

| 时间 | 操作者 | 操作 | 结果 |
|------|--------|------|------|
| 10:00 | 协调者 | git merge feature/xxx | 成功 |
| 10:15 | 项目经理 | cmake build | 成功 |
| 10:30 | 协调者 | 安装系统包 xxx | 需人类审批 |
```

## 7. 紧急情况权限

### 紧急情况定义

```
紧急情况：
- 安全漏洞需要立即修复
- 生产环境故障
- 数据丢失风险
- 需要立即部署修复
```

### 紧急情况处理

```
紧急情况发生时：
1. 协调者评估情况
2. 可以使用任何必要权限
3. 记录所有操作
4. 事后向人类汇报
```

### 紧急权限列表

```
紧急情况下，协调者可以：
- 强制推送覆盖远程
- 回滚代码
- 禁用功能
- 停止服务
- 恢复备份
```

---

## 8. 人类最终控制

### 人类保留的权力

```
即使授予了权限，人类仍然可以：
- 随时撤销权限
- 要求审批特定操作
- 查看审计日志
- 直接干预任何操作
```

### 权限撤销机制

```
人类可以通过：
- 告诉协调者撤销权限
- 修改 CLAUDE.md 中的权限配置
- 直接修改文件
```

## 9. 协调者启动时的权限声明

```
我已获得系统权限，可以在无需人类确认的情况下：
- 创建、修改、删除任何文件
- 执行 Git 操作（提交、推送、合并）
- 构建和运行项目
- 安装依赖和工具
- 配置开发环境

我将：
- 记录所有敏感操作到审计日志
- 超出权限的操作向人类请求审批
- 紧急情况使用必要权限并事后汇报
```
