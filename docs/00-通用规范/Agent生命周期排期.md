# Agent 生命周期排期

**版本**：v1.0

**用途**：基于 Agent 角色生命周期的项目排期，不是按日期，而是按阶段

---

## 1. Agent 生命周期分类

### 长期 Agent（持续存在）

| Agent | 生命周期 | 说明 |
|-------|---------|------|
| 协调者 | 全程 | 项目开始到结束，负责整体调度 |
| 项目经理 | 全程 | 项目开始到结束，负责进度管理 |
| 开发规范制定者 | 全程 | 持续跟踪规范，可能被 Fallback 唤起 |
| Reviewer | 开发阶段 | 开发期间持续进行代码审查 |
| Approver | 开发阶段 | 开发期间持续进行 PR 审核 |

### 短期 Agent（阶段式）

| Agent | 生命周期 | 触发 | 结束条件 |
|-------|---------|------|---------|
| 总架构师 | 架构阶段 | 架构设计开始 | 架构完成并审核通过 |
| 接口人 | 架构阶段 | 模块划分确定 | 接口规范确定 |
| 模块架构师 | 架构阶段 | 接口确定 | 模块架构完成 |
| 开发者 | 开发阶段 | 架构完成 | 项目发布 |

## 2. 阶段划分

### 阶段 1：规范制定

```
时长：~1周
目标：建立开发规范

活跃 Agent：
┌─────────────────────┐
│ 开发规范制定者      │ ← 主导
└─────────────────────┘
         ↓
     规范确定

其他 Agent（不活跃）：
- 协调者：监控
- 项目经理：准备任务队列
```

### 阶段 2：架构设计

```
时长：~2-4周
目标：完成整体架构设计

活跃 Agent：
┌─────────────────────┐
│ 总架构师            │ ← 主导
└─────────────────────┘
         ↓
┌─────────────────────┐
│ 接口人              │ ← 主导
└─────────────────────┘
         ↓
┌─────────────────────┐
│ 模块架构师 A        │ ← 并行
│ 模块架构师 B        │
│ 模块架构师 C        │
└─────────────────────┘

其他 Agent（不活跃）：
- 协调者：决策
- 项目经理：跟踪
- 开发规范制定者：待命
- Reviewer/Approver：未启动
```

### 阶段 3：开发实现

```
时长：~6-12个月
目标：完成功能开发

活跃 Agent：
┌─────────────────────┐
│ 开发者（多个）      │ ← 主导
└─────────────────────┘
         ↓
┌─────────────────────┐
│ 测试                 │ ← 主导
└─────────────────────┘
         ↓
┌─────────────────────┐
│ Reviewer            │ ← 主导
└─────────────────────┘
         ↓
┌─────────────────────┐
│ Approver            │ ← 主导
└─────────────────────┘

其他 Agent（持续活跃）：
- 协调者：监控、决策
- 项目经理：进度管理
- 开发规范制定者：待命（可能被 Fallback 唤起）
```

### 阶段 4：测试验收

```
时长：~1-2个月
目标：完成测试和验收

活跃 Agent：
┌─────────────────────┐
│ 测试                 │ ← 主导
└─────────────────────┘
┌─────────────────────┐
│ Reviewer            │ ← 主导
└─────────────────────┘

其他 Agent：
- 协调者：最终决策
- 项目经理：收尾管理
- Approver：最终审核
```

### 阶段 5：发布维护

```
时长：持续
目标：发布和维护

活跃 Agent：
- 协调者：监控
- 项目经理：维护任务管理
- 开发规范制定者：更新规范
- Reviewer：持续审查
- Approver：持续审核

短期 Agent 结束：
- 总架构师 ✅ 已结束
- 接口人 ✅ 已结束
- 模块架构师 ✅ 已结束
- 开发者 ✅ 已结束
```

## 3. Fallback 机制

### Fallback 定义

Fallback 是指在正常流程中遇到问题，需要唤起其他 Agent 处理的机制。

### Fallback 场景

| 场景 | 触发条件 | 唤起 Agent | 处理后 |
|------|---------|-----------|--------|
| **架构问题** | 开发遇到设计缺陷 | 架构师（总/模块） | 修复后继续开发 |
| **规范缺失** | 需要新技术/语言 | 开发规范制定者 | 补充规范后继续 |
| **技术难点** | 遇到无法解决的技术问题 | 架构师 | 重新设计后继续 |
| **依赖引入** | 需要引入第三方模块 | 开发规范制定者 | 补充规范后继续 |
| **安全/性能** | Reviewer 发现严重问题 | 安全专家/架构师 | 修复后继续 |

### Fallback 流程

```
正常流程
    ↓
检测到问题（开发者/测试/Reviewer）
    ↓
判断是否需要 Fallback
    ↓
    ├─→ 否 → 开发者自行解决
    │
    └─→ 是 → 唤起对应 Agent
            ↓
        问题修复/规范补充
            ↓
        恢复开发
```

### Fallback 优先级

```
P0 - 阻塞性问题（必须 Fallback）
- 架构设计缺陷导致无法开发
- 缺少关键规范无法继续
- 严重安全/性能问题

P1 - 重要问题（建议 Fallback）
- 多次遇到同类技术问题
- 需要引入新的技术栈
- 代码质量持续不达标

P2 - 一般问题（可不 Fallback）
- 小的实现问题，开发者可自行解决
- 文档不清晰，可以推断
```

## 4. 长期 Agent 跟踪文档

### 需要持续跟踪的 Agent

| Agent | 跟踪文档 |
|-------|---------|
| 协调者 | 迭代文档 |
| 项目经理 | 进度跟踪文档 |
| 开发规范制定者 | 规范更新日志 |
| Reviewer | 审查问题汇总 |
| Approver | 审核决策记录 |

### 规范更新日志模板

```markdown
# 开发规范更新日志

## 更新记录

### 2026-03-15 - 添加 C++ 规范
- 添加原因：引入 C++ 模块开发
- 更新内容：C++ 编码规范
- 更新人：开发规范制定者
- 审核人：协调者

### 2026-03-20 - 添加 Python 规范
- 添加原因：引入 Python 脚本模块
- 更新内容：Python 编码规范
- 更新人：开发规范制定者
- 审核人：协调者

## 当前活跃规范
- C++ 规范
- Python 规范
- Git 提交规范
- PR 规范
```

### 审查问题汇总模板

```markdown
# 代码审查问题汇总

## 问题统计
- P0 问题：X 个
- P1 问题：X 个
- P2 问题：X 个

## 常见问题模式
1. [问题模式] - 出现 X 次
   - 建议：xxx

## 修复建议
- xxx

## 趋势
- 本周问题数：X
- 上周问题数：X
- 趋势：上升/下降
```

   │            │             │            │            │

## 7. 协调者职责

### 阶段切换

```
协调者在每个阶段结束时：
1. 确认短期 Agent 交付完成
2. 确认交付物符合质量标准
3. 决定是否可以进入下一阶段
4. 记录阶段完成状态
```

### Fallback 管理

```
协调者处理 Fallback：
1. 接收 Fallback 请求
2. 评估是否需要 Fallback
3. 决定唤起哪个 Agent
4. 跟踪 Fallback 处理进度
5. 确认恢复开发
```

## 8. 排期示例

### 阶段 1：规范制定（1周）

```
Week 1:
- Day 1-2: 开发规范制定者制定代码规范
- Day 3-4: 开发规范制定者制定提交/PR规范
- Day 5: 协调者审核 → 通过
```

### 阶段 2：架构设计（3周）

```
Week 2-3:
- Day 1-3: 总架构师设计模块划分
- Day 4-7: 接口人设计模块接口
- Day 8-14: 模块架构师并行设计

Week 4:
- Day 15-18: 协调者审核架构
- Day 19-20: 根据反馈调整
```

### 阶段 3：开发实现（持续）

```
开发迭代:
- Sprint 1: 模块 A 开发 → Review → Approve
- Sprint 2: 模块 B 开发 → Review → Approve
- ...
- Fallback: 遇到架构问题 → 唤起架构师 → 修复 → 继续

长期 Agent 持续工作:
- 项目经理：跟踪进度
- Reviewer：持续审查
- Approver：持续审核
- 规范制定者：待命（Fallback 时唤起）
```

---

**此排期方案确保 Agent 团队在正确的时机做正确的事，短期 Agent 有明确的生命周期，长期 Agent 持续跟踪**
