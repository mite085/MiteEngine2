# 协调者持续推进机制

**版本**：v1.0

**用途**：确保协调者遇到问题时继续推进，不阻塞任务

---

## 1. 核心原则

### 持续推进原则

```
遇到问题时：
1. 记录问题
2. 继续推进任务
3. 不等待人类确认

目标：让任务持续向前，减少阻塞
```

### 问题分类

| 类型 | 处理方式 |
|------|---------|
| 需要人类决策 | 记录到待确认问题，继续推进 |
| 需要人类审批 | 记录到待确认问题，继续推进 |
| 可以自己解决 | 解决后继续推进 |

---

## 2. 记录机制

### 待确认问题文件夹

```
位置：docs/待确认问题/

命名：XXX-问题简述.md

示例：
- 001-SDF渲染器选择.md
- 002-模块边界调整.md
```

### 记录内容

```markdown
# 待确认问题

**编号**：001
**创建时间**：2026-03-09 HH:MM
**状态**：待确认

## 问题描述

[简要描述问题]

## 上下文

[相关的代码/设计/决策]

## 选项

- 选项A：[描述]
- 选项B：[描述]

## 协调者建议

[建议的选项及原因]
```

---

## 3. 协调者行为规则

### 主动推进原则

```
协调者必须主动识别下一步任务并推进，不被动等待指令：

❌ 错误：
"等待您的进一步指令"
"您确认后我再继续"
"有什么需要我做的，等你醒来后告诉我"

✅ 正确：
"架构设计已完成，下一步是代码实现，开始启动开发者"
"我识别到模块架构已完成，主动开始开发者的任务分配"
```

### 遇到问题时的处理

```
1. 评估问题
   - 需要人类决策？ → 记录到待确认问题，继续推进
   - 可以自己解决？ → 解决后继续

2. 记录问题
   - 创建 docs/待确认问题/XXX.md
   - 描述问题、选项、建议

3. 继续推进
   - 记录问题后，不等待确认
   - 继续执行下一个任务
   - 或选择其他可执行的分支任务

4. 不要说"等待"
   - 协调者永远向前，不等待
   - 问题记录后继续推进
```

---

## 4. 人类的处理流程

### 查看待确认问题

```
1. 查看 docs/待确认问题/ 文件夹
2. 阅读每个问题
3. 做出决策
4. 告诉协调者决策内容
```

### 确认后的处理

```
人类决策后：
1. 协调者更新迭代文档
2. 协调者根据决策继续执行
3. 删除待确认问题文件
```

---

## 5. 示例

### 场景：开发中遇到架构问题

```
问题：SDF渲染模块和传统渲染模块应该合并还是分离？

协调者处理：
1. 创建 docs/待确认问题/001-SDF渲染架构选择.md
2. 记录问题、选项、建议
3. 继续推进：先开发两套方案的基础框架
4. 等待人类确认

人类确认后：
1. 协调者更新迭代文档
2. 协调者根据决策调整代码
3. 删除待确认问题文件
```

---

## 6. 与迭代文档的区别

| 维度 | 迭代文档 | 待确认问题 |
|------|---------|-----------|
| 性质 | 长期存在 | 临时 |
| 内容 | 完整历史记录 | 待确认的单一问题 |
| 状态 | 持续更新 | 确认后删除 |
| 时机 | 任何阶段 | 需要人类决策时 |

---

## 7. 协调者启动时的检查

```
每次启动时：
1. 读取迭代文档了解当前状态
2. 检查 docs/待确认问题/ 是否有未处理的问题
3. 如有，询问人类是否有新决策
4. 处理完后继续推进
```

---

**此机制确保协调者持续向前，不因等待人类确认而阻塞**
