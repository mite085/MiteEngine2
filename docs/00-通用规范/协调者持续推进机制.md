# 协调者持续推进机制

**版本**：v1.0

**用途**：确保协调者遇到问题时继续推进，不阻塞任务

---

## 1. 核心原则

### 持续推进原则

```
遇到问题时：
1. 记录问题
2. 继续推进任务
3. 不等待人类确认

任务完成时：
1. 记录完成状态
2. 立即开始下一个待办任务
3. 不等待人类确认

目标：让任务持续向前，减少阻塞，最大化单次会话的工作量
```

### 问题分类

| 类型 | 处理方式 |
|------|---------|
| 需要人类决策 | 记录到待确认问题，继续推进 |
| 需要人类审批 | 记录到待确认问题，继续推进 |
| 可以自己解决 | 解决后继续推进 |

---

## 2. 记录机制

### 待确认问题文件夹

```
位置：docs/待确认问题/

命名：XXX-问题简述.md

示例：
- 001-SDF渲染器选择.md
- 002-模块边界调整.md
```

### 记录内容

```markdown
# 待确认问题

**编号**：001
**创建时间**：2026-03-09 HH:MM
**状态**：待确认

## 问题描述

[简要描述问题]

## 上下文

[相关的代码/设计/决策]

## 选项

- 选项A：[描述]
- 选项B：[描述]

## 协调者建议

[建议的选项及原因]
```

## 3. 协调者行为规则

### 主动推进原则

```
协调者必须主动识别下一步任务并立即推进：

✔ 正确行为：
"架构设计已完成，开始执行下一个任务：第三方库评审"
"模块A架构已完成，并行启动模块B架构设计"
"开发者产出已收到，启动测试 Agent 进行验证"

✖ 错误行为（绝对禁止）：
"等待您的进一步指令"
"您确认后我再继续"
"有什么需要我做的，等你醒来后告诉我"
"以上是我的分析，请确认"
"请问您希望我接下来做什么"
"以上工作已完成，请审阅"

产出落盘规则（绝对禁止只输出不保存）：
- Agent 产出的任何文档、代码、报告，必须调用 Write/Edit 工具写入文件
- 禁止只在对话中输出内容而不保存到磁盘
- 每次 Agent 完成任务后，协调者必须验证产出文件是否存在
- 验证方式：用 Read 或 ls 确认文件已创建且内容非空

停止条件（仅以下情况才停下来等人类）：
1. 任务队列中所有任务全部完成
2. 所有剩余任务都处于阻塞状态且没有可并行的替代任务
3. 遇到必须由人类决策的问题且无法绕行
```

### 遇到问题时的处理

```
1. 评估问题
   - 需要人类决策？ → 记录到待确认问题，继续推进
   - 可以自己解决？ → 解决后继续

2. 记录问题
   - 创建 docs/待确认问题/XXX.md
   - 描述问题、选项、建议

3. 继续推进
   - 记录问题后，不等待确认
   - 继续执行下一个任务
   - 或选择其他可执行的分支任务

4. 不要说"等待"
   - 协调者永远向前，不等待
   - 问题记录后继续推进
```

## 4. 人类的处理流程

### 查看待确认问题

```
1. 查看 docs/待确认问题/ 文件夹
2. 阅读每个问题
3. 做出决策
4. 告诉协调者决策内容
```

### 确认后的处理

```
人类决策后：
1. 协调者更新迭代文档
2. 协调者根据决策继续执行
3. 删除待确认问题文件
```

---

## 5. 示例

### 场景：开发中遇到架构问题

```
问题：SDF渲染模块和传统渲染模块应该合并还是分离？

协调者处理：
1. 创建 docs/待确认问题/001-SDF渲染架构选择.md
2. 记录问题、选项、建议
3. 继续推进：先开发两套方案的基础框架
4. 等待人类确认

人类确认后：
1. 协调者更新迭代文档
2. 协调者根据决策调整代码
3. 删除待确认问题文件
```

## 6. 与迭代文档的区别

| 维度 | 迭代文档 | 待确认问题 |
|------|---------|-----------|
| 性质 | 长期存在 | 临时 |
| 内容 | 完整历史记录 | 待确认的单一问题 |
| 状态 | 持续更新 | 确认后删除 |
| 时机 | 任何阶段 | 需要人类决策时 |

---

## 7. 协调者启动时的检查

```
每次启动时：
1. 读取迭代文档了解当前状态
2. 检查 docs/待确认问题/ 是否有未处理的问题
3. 如有，询问人类是否有新决策
4. 处理完后继续推进
```

## 8. 阻塞降级策略

### 背景

MiniMax Token 每 5 小时刷新一次，不累积。协调者阻塞 = Token 浪费。必须最大化每个周期内的 Token 利用率。

### 降级流程

```
正常推进 → 遇到阻塞
                ↓
          记录待确认问题
                ↓
          检查任务队列中是否有可并行的非阻塞任务
                ↓
          ├→ 有 → 切换执行非阻塞任务
          └→ 无 → 执行填充任务池中的任务
                    ↓
               填充任务也做完了 → 才真正停止等待人类
```

### 填充任务定义

当所有主线任务都阻塞时，协调者从填充任务池中选取任务执行：

```
填充任务优先级（从高到低）：
1. 已有代码的测试补充（提高覆盖率）
2. 已有文档的一致性审查
3. 已完成模块的性能审计
4. 代码规范合规性检查
5. 迭代文档和决策记录的整理归档
```

填充任务池由协调者在 CLAUDE.md「当前任务」章节维护，根据项目阶段动态更新。

### 规则

```
1. 填充任务不能引入新的人类决策点
2. 填充任务不能修改主线架构或接口
3. 填充任务的产出记录到迭代文档
4. 填充任务结束后，检查阻塞是否已解除
```

## 9. 批量决策请求

### 背景

每次阻塞等待人类都可能浪费一个 Token 周期。协调者应提前识别需要人类决策的节点，批量收集后一次性请求。

### 机制

```
推进过程中：
1. 识别未来 2-3 步可能需要人类决策的节点
2. 将这些决策点提前收集，附上协调者建议
3. 在一次停止中一并提交给人类

格式：
━━━━━━━━━━━━━━━━━━━━━━━━
待确认问题汇总（共 N 项）
━━━━━━━━━━━━━━━━━━━━━━━━
1. [问题A] → 建议：选项X
2. [问题B] → 建议：选项Y
3. [问题C] → 建议：选项Z
━━━━━━━━━━━━━━━━━━━━━━━━

规则：
- 不要一个问题停一次，攒够再停
- 如果问题之间有依赖，标注依赖关系
- 每个问题必须附带协调者建议（由对应Agent提供）
```

---

**此机制确保协调者持续向前，最大化 Token 利用率，不因等待人类确认而浪费资源**
