# Agent 代码提交流程

**版本**：v1.0

**用途**：规范 Agent 团队的代码提交流程，包括 PR 提交、MR 合并、代码审查

---

## 流程概览

> **适用角色**：所有角色（了解整体流程）

```
开发者 Agent 完成代码
       ↓
Reviewer Agent 对抗式审查
       ↓
       ├─→ 审查通过 → 提交 PR
       │         ↓
       │      审核 Agent 审核 PR
       │         ↓
       │      PR 通过 → 合并到主分支
       │         ↓
       │      代码发布
       │
       └─→ 审查不通过 → 退回开发者修复 → 重新审查
```

---

## 角色定义

> **适用角色**：所有角色

### 1. 开发者 Agent (Developer)

```
职责：
- 实现功能代码
- 确保代码可编译
- 编写基本的单元测试
- 提交代码供审查

输出：
- 可编译的代码
- 单元测试代码
- 代码变更说明
```

### 2. 审查 Agent (Reviewer)

```
职责：
- 对抗式代码审查
- 发现潜在问题
- 提供修复建议
- 确保代码质量

输出：
- 审查报告
- 问题列表（按严重程度）
- 通过/不通过结论
```

### 3. 审核 Agent (Approver)

```
职责：
- 审核 PR 的整体质量
- 检查代码变更的合理性
- 确认符合项目规范
- 决定是否合并

输出：
- 审核报告
- 合并/不合并结论
- 合并前需要满足的条件
```

## 对抗式代码审查

> **适用角色**：Reviewer（必读）、开发者（了解审查标准）

### 什么是对抗式审查

对抗式审查不仅仅是"找问题"，而是以**攻击者的思维**审视代码：
- 如果我是黑客，这个代码有什么漏洞？
- 如果输入极端情况，代码会崩溃吗？
- 这个实现是否有隐藏的边界问题？
- 是否有性能陷阱？

### 审查维度

| 维度 | 审查内容 | 对抗性问题示例 |
|------|---------|--------------|
| **安全性** | 输入验证、权限控制 | 缓冲区溢出、SQL注入、权限绕过 |
| **正确性** | 逻辑是否正确 | 边界条件、并发问题、状态不一致 |
| **健壮性** | 异常处理 | 未捕获异常、资源泄漏、空指针 |
| **性能** | 性能影响 | 内存泄漏、算法复杂度、循环效率 |
| **可维护性** | 代码质量 | 重复代码、命名混乱、注释不足 |
| **测试覆盖** | 测试完整性 | 边界测试、异常测试、并发测试 |

### 审查强度分级

| 级别 | 问题类型 | 处理方式 |
|------|---------|---------|
| **P0 - 致命** | 安全漏洞、崩溃bug | 必须修复，否则拒绝合并 |
| **P1 - 严重** | 逻辑错误、内存泄漏 | 必须修复 |
| **P2 - 中等** | 性能问题、代码异味 | 建议修复 |
| **P3 - 轻微** | 格式问题、注释不足 | 可选修复 |

## PR 提交流程

> **适用角色**：开发者（§步骤1/3）、Reviewer（§步骤2）、Approver（§步骤4）

### 步骤 1：开发者提交代码

```
开发者 Agent 完成代码后：
1. 确保代码可编译
2. 运行单元测试
3. 编写变更说明
4. 创建分支
5. 提交代码
6. 创建 PR
```

### 步骤 2：Reviewer 审查

```
审查 Agent 收到 PR 后：
1. 拉取代码到本地
2. 进行对抗式审查
3. 输出审查报告
4. 给出结论：通过/需要修改
```

### 步骤 3：开发者修复

```
如果审查不通过：
1. 开发者 Agent 修复问题
2. 提交修复
3. 重新发起审查
```

### 步骤 4：Approver 审核

```
审查通过后：
1. 审核 Agent 检查 PR
2. 确认变更符合项目目标
3. 决定是否合并
4. 执行合并（如通过）
```

## 提交规范

> **适用角色**：开发者（必读）

### 提交触发条件（强制要求）

```
开发者 Agent 在以下情况必须立即提交代码：

1. 完成任何一个独立的小功能后
   - 例如：实现了一个数学函数、完成了一个简单的类
   - 不要等到"大功能"完成再提交

2. 完成阶段性里程碑后
   - POC 骨架代码完成后
   - 模块接口定义完成后
   - 编译验证通过后

3. 每日工作结束前
   - 即使代码不完整，也应提交当日进度
   - 使用明确的 commit message 说明进度

4. 遇到阻塞需要换任务时
   - 当前任务阻塞，切换前必须提交已有进度
   - 禁止留下未提交的代码切换任务
```

### 小步提交激励机制

```
为什么必须小步提交：

1. 降低风险
   - 大块代码难以审查，容易遗漏问题
   - 小步提交便于 Reviewer 逐个检查

2. 便于回滚
   - 如果发现 bug，可以精确回滚到有问题的 commit
   - 大块代码回滚会影响其他功能

3. 进度透明
   - 每次 commit 都能看到实际进展
   - 协调者可以准确了解项目状态

4. 心理激励
   - 频繁提交带来成就感
   - 避免"写了很久但看不到成果"的感觉

原则：宁可多提交，不要少提交
      提交5次100行代码，优于1次提交500行代码
```

### 分支命名规范

```
格式：[类型]/[功能]-[Ticket编号]

示例：
- feature/rendering-sdf-module
- fix/memory-leak-issue-123
- refactor/physics-system-cleanup
```

### Commit 颗粒度规范（强制要求）

```
原则：小步 commit，每次只做一件事

规则：
1. 单次 commit 不应涉及超过 5 个文件的更改
2. 每个 commit 应有明确的功能/修复目的
3. 相关的改动应放在一起，不相关的应拆分

例外情况：
- 如果开发者认为虽然涉及 5+ 文件，但属于同一逻辑单元不可拆分
- 这种情况需要在 commit message 中说明原因

示例（好的 commit）：
- feat: Add SDF renderer class (3 files)
- fix: Fix memory leak in buffer (2 files)
- docs: Update README (1 file)

示例（应拆分的 commit）：
- ❌ feat: Add renderer + physics + input (10 files)
  → 应该拆分成 3 个 commit
```

### PR 标题规范

```
格式：[类型] [简短描述] (#Ticket)

示例：
- [Feature] Add SDF rendering module (#1)
- [Fix] Fix memory leak in renderer (#2)
- [Refactor] Clean up physics system (#3)
```

### Commit 消息规范（强制要求）

```
格式：<英文标题>

<中文描述>

示例：
Add SDF raymarching renderer

添加 SDF 光线步进渲染器，支持基础球体、立方体和布尔运算

---

规则：
1. 标题：使用英文，首字母大写，简洁明了
2. 描述：使用中文，详细说明变更内容
3. 标题和描述之间空一行
4. 描述可省略（如变更简单明了）
5. 使用 git commit -m "标题" -m "描述" 格式
```

### PR 描述模板

```markdown
## 变更内容
[描述本次变更做了什么]

## 变更原因
[为什么需要这个变更]

## 测试情况
- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 手动测试通过（如适用）

## 相关 Issue
Closes #123
```

## 合并规则

> **适用角色**：Approver（必读）

### 需要满足的条件

```
合并前必须满足：
□ 代码审查通过（至少1个 Reviewer 批准）
□ PR 审核通过（Approver 批准）
□ 所有测试通过
□ 无冲突
□ 分支是最新的
```

### 合并方式

| 方式 | 使用场景 | 命令 |
|------|---------|------|
| Squash Merge | 保持线性历史 | git merge --squash |
| Merge Commit | 保留完整历史 | git merge |
| Rebase | 保持线性且干净 | git rebase |

推荐使用 **Squash Merge**，保持主分支线性整洁。

## 审核 Agent 审查清单

> **适用角色**：Approver（必读）

### PR 整体评估

| 检查项 | 评估 |
|--------|------|
| 变更是否符合项目目标 | □ 是 / □ 否 |
| 变更范围是否合理 | □ 是 / □ 否 |
| 是否有必要的测试 | □ 是 / □ 否 |
| 是否有安全风险 | □ 是 / □ 否 |
| 是否符合编码规范 | □ 是 / □ 否 |

### 决策

```
□ 批准合并
□ 需要修改后合并
□ 拒绝合并
□ 暂不处理
```

## 协调者职责

> **适用角色**：协调者（必读）

### 核心原则：推动提交是协调者的首要责任

```
协调者必须时刻关注代码提交状态，确保：
1. 代码不会长时间停留在本地未提交
2. 每个阶段性成果都及时落盘
3. 提交行为是主动的，不是被动的
```

### 在提交流程中，协调者负责：

```
1. 任务分配
   - 将代码提交任务分配给 Developer
   - 将审查任务分配给 Reviewer
   - 将审核任务分配给 Approver

2. 提交催办（关键职责）
   - 定期检查未提交的代码进度
   - POC/模块完成后，立即要求 Developer 提交
   - 每日至少提醒一次提交状态
   - 遇到阻塞时，确保阻塞前已提交进度

3. 流程控制
   - 监控流程进度
   - 处理阻塞
   - 确保流程完整

4. 异常处理
   - 处理审查不通过
   - 处理合并冲突
   - 处理审核拒绝

5. 结果记录
   - 记录审查结果
   - 记录合并结果
   - 更新任务状态
```

### 协调者提交检查清单

```
每次与 Developer 交互时，检查以下事项：

□ 上次提交是什么时候？
□ 距离现在已经完成了多少新工作？
□ 是否有未提交的代码超过 200 行？
□ 是否到了里程碑节点需要提交？
□ 开发者是否在等待任务而没有提交？

如果有任何一项为"是"，立即要求提交。
```

## 经验反馈

### 常见审查问题

| 问题类型 | 常见问题 | 建议 |
|---------|---------|------|
| 安全性 | 未验证输入 | 添加输入验证 |
| 错误处理 | 捕获异常但不处理 | 添加错误处理或日志 |
| 性能 | 循环中分配内存 | 移到循环外 |
| 并发 | 非线程安全访问 | 添加锁或使用原子操作 |
| 测试 | 无边界测试 | 添加边界测试用例 |
