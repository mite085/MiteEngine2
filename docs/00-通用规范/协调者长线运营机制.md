# 协调者长线运营机制

**版本**：v1.0

**用途**：确保协调者能够长时间持续运行，处理阻塞和token耗尽场景

---

## 核心挑战

Claude Code 在长时间运行中会遇到：
1. **上下文耗尽** - Token数量达到上限
2. **任务阻塞** - 某个任务无法继续
3. **Token耗尽** - 额度用完，无法继续

---

## 能力 1：任务阻塞处理

### 阻塞定义

```
以下情况视为任务阻塞：
- 依赖外部资源（等待人类决策、等待外部API）
- 技术方案不可行（需要重新设计）
- 需求不明确（需要苏格拉底询问）
- 遇到未知错误（无法自行解决）
```

### 阻塞处理流程

```
检测到阻塞
    ↓
记录阻塞详情到迭代文档
    ↓
分析阻塞任务与其他任务的依赖关系
    ↓
    ├─→ 有依赖 → 记录阻塞，暂停相关任务
    │
    └─→ 无依赖 → 切换到不相关的任务继续执行
            ↓
        继续处理其他任务
            ↓
        定期检查阻塞任务是否解除
```

### 任务依赖分析

```
分析维度：
1. 数据依赖 - 输出是否被其他任务使用
2. 上下文依赖 - 是否需要前面的结果
3. 资源依赖 - 是否需要相同资源

示例：
- 任务A：设计渲染模块 → 任务B：实现渲染模块 → 任务C：测试渲染模块
  → A阻塞 → B和C都无法执行

- 任务A：设计渲染模块 → 任务B：设计物理模块
  → A阻塞 → B可以独立执行
```

### 阻塞切换规则

```
切换策略：
1. 优先切换到无依赖的待办任务
2. 如果所有任务都有依赖，记录阻塞并报告用户
3. 切换时更新迭代文档的任务状态

示例：
- 当前阻塞：渲染架构设计
- 可切换：物理架构设计、编码规范制定
- 选择：物理架构设计（不依赖渲染）
```

## 能力 2：Token 耗尽处理

### Token 耗尽检测

```
检测方式：
- 尝试发送消息时收到错误
- 收到 "context window exceeded" 或类似错误
- 收到 "rate limit" 或 "quota exceeded" 错误
```

### Token 耗尽处理流程

```
检测到Token耗尽
    ↓
保存当前任务状态到迭代文档
    ↓
记录"待恢复"标记（包含：当前任务、进度、下一步动作）
    ↓
告知用户当前状态
    ↓
（用户开启新会话后自动恢复）
```

### 恢复执行流程

```
用户开启新会话
    ↓
读取迭代文档
    ↓
确认任务状态（待恢复/进行中/阻塞）
    ↓
输出简要恢复摘要（不超过3行）
    ↓
立即继续执行任务（不等待用户确认）
```

### 恢复机制

```
恢复规则：
- 协调者无法自动轮询，依赖用户开启新会话触发恢复
- 恢复后立即继续执行，不等待用户确认
- 恢复摘要控制在3行以内，避免浪费上下文

实现方式：
1. Token耗尽时保存完整状态到迭代文档
2. 用户开启新会话后，协调者读取迭代文档
3. 立即从中断点继续执行
```

## 能力 3：上下文管理

### 上下文耗尽预防

```
预防策略：
1. 定期清理不必要的中间状态
2. 只保留关键上下文信息
3. 将重要信息沉淀到文件

清理时机：
- 每个任务完成后
- 上下文使用超过50%时
- 切换任务时
```

### 上下文重建

```
当开启新对话时：
1. 读取迭代文档获取任务状态
2. 读取项目配置获取约束
3. 读取最近的任务记录
4. 立即从中断点继续执行（不等待用户确认）
```

## 协调者 Rules

### R-Long-1：阻塞处理规则

```
当检测到任务阻塞时：
1. 立即记录到迭代文档：
   - 阻塞任务名称
   - 阻塞原因
   - 阻塞时间
   - 建议解决方案

2. 分析依赖关系：
   - 检查其他任务是否依赖此任务
   - 标记相关任务状态

3. 切换任务：
   - 优先选择无依赖的任务
   - 更新任务队列
   - 向用户报告阻塞和切换
```

### R-Long-2：Token 耗尽规则

```
当检测到 Token 耗尽时：
1. 保存当前状态到迭代文档
2. 记录任务进度
3. 告知用户当前状态
4. 说明将在用户再次发起时继续

当用户再次发起时：
1. 读取迭代文档
2. 恢复任务状态
3. 继续执行
```

### R-Long-3：上下文管理规则

```
上下文使用超过50%时：
1. 识别可以清理的内容
2. 将关键信息沉淀到文件
3. 清理中间状态

上下文接近耗尽时：
1. 提前告知用户
2. 保存完整状态
3. 准备重建上下文
```

## 任务状态管理

### 状态定义

| 状态 | 含义 | 后续动作 |
|------|------|---------|
| `待开始` | 尚未开始 | 可执行 |
| `进行中` | 正在执行 | 继续执行 |
| `已完成` | 执行完成 | 记录结果 |
| `阻塞` | 无法继续 | 切换任务 |
| `待恢复` | Token耗尽，等待恢复 | 用户发起后恢复 |

### 状态更新规则

```
任务状态变更时：
1. 记录变更原因
2. 更新任务队列
3. 如需切换，标记原任务为阻塞
4. 向用户报告状态变更
```

## 迭代文档记录格式

### 阻塞记录

```markdown
## 阻塞记录

### [任务名称]
- 阻塞时间：[时间]
- 阻塞原因：[原因]
- 建议方案：[方案]
- 依赖任务：[受影响的任务列表]
- 切换任务：[切换到的任务]
```

### 待恢复记录

```markdown
## Token 耗尽记录

### [任务名称]
- 耗尽时间：[时间]
- 任务进度：[进度描述]
- 恢复后动作：[下一步]
- 用户发起时间：[用户重新发起的时间]
```

## 协调者启动检查清单

每次任务开始前，检查：

```
□ 1. 读取迭代文档，确认任务状态
□ 2. 检查是否有"待恢复"任务
□ 3. 检查是否有阻塞任务
□ 4. 分析当前任务与其他任务的依赖
□ 5. 立即开始执行（不等待用户确认）
```

**此机制确保协调者在长时间运行中能够稳定工作，有效处理各类异常情况**
