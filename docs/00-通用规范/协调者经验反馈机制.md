# 协调者经验反馈机制

**版本**：v1.0

**用途**：在开发过程中总结经验，反馈到协调者体系建设中

---

## 核心概念

### 什么是经验反馈

经验反馈是指：
- 每次任务完成后进行复盘
- 提取有价值的经验教训
- 将经验转化为可复用的规范
- 持续优化协调者体系

### 为什么需要

```
一次性建设的规范不可能完美
通过实践中的经验不断迭代
才能让协调者体系越来越强大
```

---

## 经验分类

### 类型 1：流程经验

```
定义：关于任务执行流程的经验

示例：
- "架构师的输出格式不统一，导致开发者理解困难"
- "任务分解时遗漏了测试环节"
- "并行任务分配不合理，等待时间过长"
```

### 类型 2：协作经验

```
定义：关于 Agent 之间协作的经验

示例：
- "架构师的文档太技术化，开发者难以理解"
- "Reviewer 的反馈不够具体，修复效率低"
- "测试 Agent 生成的测试用例难以运行"
```

### 类型 3：质量经验

```
定义：关于质量把控的经验

示例：
- "代码可编译不代表功能正确，需要功能验证"
- "架构设计缺少性能评估，后续重构成本高"
- "单元测试覆盖率高但集成测试缺失"
```

### 类型 4：风险经验

```
定义：关于风险识别的经验

示例：
- "技术选型时未考虑移动端兼容性"
- "时间估计过于乐观，实际耗时是预期的2倍"
- "依赖的第三方库有许可证风险"
```

## 经验提取流程

### 步骤 1：任务结束复盘

```
每次任务完成后，协调者应进行复盘：

复盘问题：
1. 任务目标是否达成？
2. 过程中遇到了什么问题？
3. 问题的根本原因是什么？
4. 有什么可以改进的地方？
```

### 步骤 2：经验分类

```
将提取的经验归类：
- 流程经验 → 更新工作规范
- 协作经验 → 更新 Agent Prompt
- 质量经验 → 更新质量标准
- 风险经验 → 更新风险清单
```

### 步骤 3：评估价值

```
评估每个经验的价值：

| 维度 | 问题 |
|------|------|
| 普遍性 | 其他任务也可能遇到？ |
| 影响程度 | 对任务完成影响多大？ |
| 可操作性 | 能否转化为具体规范？ |

只有有价值的经验才反馈到体系
```

### 步骤 4：更新规范

```
将高价值经验转化为规范：

1. 写入当前协调者迭代文档
2. 标记需要更新的目标规范
3. 定期（每周/每月）汇总更新到通用规范
```

## 反馈输出格式

### 每次任务后的经验记录

```markdown
## 经验记录

### 任务：[任务名称]

#### 提取的经验
| 类型 | 描述 | 价值评估 |
|------|------|---------|
| 流程 | xxx | 高/中/低 |
| 协作 | xxx | 高/中/低 |

#### 建议更新
- [规范名称] → [具体更新内容]
```

### 定期汇总报告

```markdown
## 经验汇总报告

**周期**：2026年3月第1周

### 经验统计
- 流程经验：X个
- 协作经验：X个
- 质量经验：X个
- 风险经验：X个

### 高价值经验
1. [经验描述]
   → 建议更新到 [规范名称]

2. [经验描述]
   → 建议更新到 [规范名称]

### 已更新规范
- [规范名称] → 更新内容
```

## 协调者使用规范

### R-Feed-1：复盘时机

```
每个任务完成后必须复盘：
- 架构设计任务完成
- 代码开发任务完成
- 测试任务完成
- 审查任务完成
```

### R-Fed-2：记录要求

```
每次复盘必须记录：
1. 任务目标 vs 实际结果
2. 遇到的主要问题
3. 根本原因分析
4. 可复用的经验
```

### R-Feed-3：更新触发

```
满足以下条件时应更新通用规范：
1. 同一类型经验出现 3 次以上
2. 经验有明确的改进建议
3. 改进建议经过验证有效
```

### R-Feed-4：更新流程

```
规范更新流程：
1. 先记录到迭代文档
2. 在汇总报告中标记
3. 定期（如每周）汇总更新
4. 更新后记录变更日志
```

## 经验库（示例）

### 流程经验库

| 经验 | 建议更新 |
|------|---------|
| 任务分解粒度不一致 | Agent Prompt 模板中明确粒度标准 |
| 缺少中间检查点 | 工作规范增加检查点 |
| 任务交接信息不完整 | 规范交接清单 |

### 协作经验库

| 经验 | 建议更新 |
|------|---------|
| 架构输出过于技术化 | 架构师 Prompt 增加"输出面向开发者"要求 |
| 审查反馈不够具体 | Reviewer Prompt 增加"问题描述模板" |
| 测试用例难以运行 | 测试 Agent Prompt 增加"可运行性检查" |

### 质量经验库

| 经验 | 建议更新 |
|------|---------|
| 可编译≠功能正确 | 质量标准增加"功能验证"要求 |
| 架构缺少性能评估 | 架构师 Prompt 增加"性能考量"章节 |
| 集成测试缺失 | 工作规范增加"测试完整性检查" |

### 风险经验库

| 经验 | 建议更新 |
|------|---------|
| 第三方库风险 | 任务启动前增加"依赖审查"步骤 |
| 时间估计偏差大 | 任务分解增加"时间缓冲"建议 |
| 技术选型欠考虑 | 增加"技术选型评估"环节 |

## 持续优化机制

### 短期（每次任务）
```
复盘 → 记录经验 → 更新迭代文档
```

### 中期（每周）
```
汇总一周经验 → 识别高价值经验 → 标记待更新规范
```

### 长期（每月）
```
审视规范更新 → 验证改进效果 → 调整规范内容
```

## 使用示例

### 示例：架构设计任务后的经验反馈

**任务完成后的复盘**：

```
1. 目标：设计渲染系统架构
2. 实际：完成了，但模块划分不合理
3. 问题：渲染模块和窗口管理耦合
4. 经验：架构设计应先确定模块边界
```

**经验记录**：

```markdown
## 经验记录

### 任务：渲染系统架构设计

#### 提取的经验
| 类型 | 描述 | 价值 |
|------|------|------|
| 流程 | 架构设计缺少模块边界确认环节 | 高 |

#### 建议更新
- 架构师 Prompt 模板 → 增加"模块边界确认"步骤
```
