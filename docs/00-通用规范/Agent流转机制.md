# Agent 流转机制

**版本**：v1.0

**用途**：细化 Agent 的启用时机、输入信息、职责边界

---

## 1. Agent 启用时机

### 阶段 vs Agent 映射

| 阶段 | 启用 Agent | 触发条件 |
|------|-----------|---------|
| **项目启动** | 开发规范制定者 | 项目初始化时 |
| **架构设计** | 总架构师 | 开始架构设计时 |
| **架构设计** | 接口人 | 总架构师完成模块划分后 |
| **架构设计** | 模块架构师 | 接口人确定接口后（可并行） |
| **第三方库评审** | 总架构师 | 架构设计完成后，产出库评审清单 |
| **架构验证** | 开发者 | 人类确认库清单后，PoC骨架+接口编译验证 |
| **开发阶段** | 开发者 | 架构验证通过后 |
| **开发阶段** | 测试 | 开发者完成代码后 |
| **开发阶段** | Reviewer | 测试通过后 |
| **发布阶段** | Approver | Reviewer 通过后 |

### 任务粒度映射

| 任务类型 | 启用 Agent | 说明 |
|---------|-----------|------|
| 制定规范 | 开发规范制定者 | 代码/PR/文档规范 |
| 技术选型 | 总架构师 | 技术方案决策 |
| 模块设计 | 模块架构师 | 单个模块的详细设计 |
| 接口设计 | 接口人 | 模块间接口协调 |
| 功能实现 | 开发者 | 具体代码编写 |
| 测试编写 | 测试 | 单元/集成测试 |
| 代码审查 | Reviewer | 代码质量审查 |
| PR审核 | Approver | 合并决策 |

## 2. Agent 输入信息

### 通用输入（所有 Agent 必须接收）

```
1. 任务描述
   - 要做什么
   - 验收标准
   - 限制条件

2. 项目约束
   - 技术约束
   - 时间约束
   - 质量要求

3. 上下文信息
   - 需求文档引用
   - 已有决策记录
   - 相关规范的引用
```

### 各 Agent 特定输入

#### 总架构师

```
输入：
- 需求迭代文档
- 项目配置（技术约束）
- 终极目标

不输入：
- 具体模块细节（由模块架构师负责）
```

#### 接口人

```
输入：
- 总架构师的模块划分方案
- 各模块架构师的接口定义
- 整体功能要求

职责：
- 协调接口，不是从头设计
```

#### 模块架构师

```
输入：
- 总架构师的模块划分
- 模块功能需求
- 技术选型决策

不输入：
- 其他模块的内部设计
```

#### 开发者

```
输入：
- 模块架构文档
- 编码规范
- 接口定义

不输入：
- 架构设计思路（不需要知道为什么）
```

#### 测试

```
输入：
- 模块架构文档
- 开发者代码
- 测试框架规范

不输入：
- 实现细节（只测功能）
```

#### Reviewer

```
输入：
- 开发者代码
- 编码规范
- 安全/性能要求

不输入：
- 业务背景（只审查代码质量）
```

#### Approver

```
输入：
- PR 内容
- Reviewer 审查报告
- 项目目标

职责：
- 审核整体，不审核细节
```

## 3. 职责边界

### 边界原则

```
1. 不越权
   - 只做职责范围内的事
   - 需要其他角色做的事 → 协调者/项目经理处理

2. 不推责
   - 职责范围内的事必须完成
   - 遇到问题 → 记录并上报，不是跳过

3. 有输出
   - 每个 Agent 必须有明确的产出
   - 产出必须符合质量标准
```

### 各 Agent 职责边界

#### 总架构师

```
✅ 职责内：
- 制定模块划分方案
- 做出技术选型决策
- 定义设计原则
- 审核模块架构

❌ 职责外：
- 具体模块的内部设计
- 代码实现
- 接口细节（由接口人负责）
```

#### 接口人

```
✅ 职责内：
- 设计模块间接口
- 审查依赖关系
- 确保模块解耦
- 协调模块架构师

❌ 职责外：
- 模块内部架构
- 代码实现
- 技术选型（由总架构师负责）
```

#### 模块架构师

```
✅ 职责内：
- 设计模块内部架构
- 定义模块接口（供外部使用）
- 审核开发者代码是否符合架构

❌ 职责外：
- 其他模块的设计
- 整体架构决策
- 接口协调（由接口人负责）
```

#### 开发者

```
✅ 职责内：
- 按照架构文档实现功能
- 编写单元测试
- 确保代码可编译

❌ 职责外：
- 架构设计
- 技术选型
- 代码审查
```

#### 测试

```
✅ 职责内：
- 编写测试用例
- 验证功能正确性
- 报告测试结果

❌ 职责外：
- 修复代码（发现bug → 退回开发者）
- 架构设计
- 代码审查
```

#### Reviewer

```
✅ 职责内：
- 代码质量审查
- 发现问题
- 给出修复建议

❌ 职责外：
- 修复代码（发现问题 → 退回开发者）
- 决定是否合并（由Approver负责）
- 架构设计
```

#### Approver

```
✅ 职责内：
- 审核 PR 整体质量
- 决定是否合并
- 确保符合项目目标

❌ 职责外：
- 代码细节审查（由Reviewer负责）
- 具体实现
- 技术选型
```

## 4. 流转流程

### 标准流程

```
人类指令 → 协调者
              ↓
         任务分析
              ↓
         选择Agent
              ↓
         准备输入 ← 从迭代文档获取上下文
              ↓
         启动Agent
              ↓
         Agent执行
              ↓
         收集输出
              ↓
         质量检查
              ↓
         ├─→ 通过 → 分配下一任务
         │
         └─→ 不通过 → 退回修改
```

### 并行流程

```
                    → 模块架构师A
                   /
总架构师 → 接口人 ──→ 模块架构师B
                   \
                    → 模块架构师C
```

### 反馈流程

```
开发者 → 测试 → 不通过 → 退回开发者

开发者 → Reviewer → 不通过 → 退回开发者

所有审查 → Approver → 不通过 → 退回修改
```

## 5. 信息传递规范

### 任务交接清单

| 从 | 到 | 必须传递 |
|----|----|---------|
| 协调者 | 任何Agent | 任务描述、验收标准、项目约束 |
| 总架构师 | 接口人 | 模块划分方案 |
| 总架构师 | 模块架构师 | 模块划分、技术选型 |
| 接口人 | 模块架构师 | 接口定义 |
| 模块架构师 | 开发者 | 模块架构文档、接口规范 |
| 开发者 | 测试 | 代码、接口定义 |
| 测试 | Reviewer | 测试报告、代码 |
| Reviewer | Approver | 审查报告 |
| 任何Agent | 协调者 | 产出物、问题报告 |

### 信息格式

```
任务交接格式：

## 任务信息
[任务描述]

## 输入信息
- [输入1]
- [输入2]

## 约束条件
- [约束1]
- [约束2]

## 验收标准
- [标准1]
- [标准2]

## 输出要求
[产出格式要求]
```

## 6. 信息传递与需求澄清

### 6.1 信息传递机制

```
任务交接流程：

发送方 Agent：
1. 按照任务交接清单准备信息
2. 确认信息完整后发送
3. 记录发送内容

接收方 Agent：
1. 验证信息完整性
2. 如有不明确，使用苏格拉底式提问
3. 确认理解正确后开始执行
```

### 6.2 需求澄清流程

```
当接收方 Agent 收到任务，发现需求不明确时：

1. 使用苏格拉底式提问明确需求
   - 澄清问题：确保理解正确
   - 探查原因：理解决策背景

2. 记录问答内容
   - 问题
   - 回答
   - 澄清后的需求

3. 如仍无法明确

         ↓
    报告给项目经理

项目经理：
1. 能解答 → 解答后继续执行
2. 不能解答 → 尝试联系架构师/规范制定者

架构师/规范制定者：
1. 能解答 → 解答后继续执行
2. 不能解答 → 报告协调者

协调者：
（不尝试解答问题）
1. 整理问题
2. 向人类汇报，等待人类指令
```

### 6.3 逐级上报机制

```
上报路径：

执行 Agent → 项目经理 → 架构师/规范制定者 → 协调者 → 人类

注意：
- 协调者不解答技术问题
- 协调者只负责向人类汇报，等待人类指令
- 协调者必须保持上下文干净
```

上报条件：
- 需求不明确且无法通过提问澄清
- 遇到技术问题且无法自行解决
- 发现职责外的重大问题
- 需要做超出职责范围的决策

上报内容：
- 问题描述
- 已尝试的解决方案
- 需要上级提供的支持
```

### 6.4 无需单独"话事人"

```
本方案不使用单独的"话事人"角色，而是：
- 发送方：按清单准备信息（已有规范）
- 接收方：验证+提问（苏格拉底式）
- 项目经理：信息完整性确认 + 初步问题解决
- 协调者：最终决策 + 人类汇报

这确保了信息传递的完整性，同时保持了角色职责的清晰
```

## 7. 异常处理

### 边界情况

| 情况 | 处理 |
|------|------|
| Agent 无法完成任务 | 记录问题 → 报告协调者 |
| Agent 之间有冲突 | 协调者裁决 |
| 输入信息不完整 | 协调者补充 → 不是自己猜 |
| 发现职责外的问题 | 记录 → 报告协调者，不是忽略 |

### 协调者介入条件

```
需要协调者介入：
- Agent 遇到无法解决的问题
- Agent 之间有冲突
- 需要做职责外的决策
- 任务需要重新分配
```

## 8. 第三方库评审流程

### 触发时机

架构设计完成后，进入开发阶段前。

### 流程

```
架构设计完成
    ↓
总架构师产出《第三方库评审清单》
    ↓
协调者将清单提交给人类审阅（停止条件）
    ↓
人类逐项确认
    ↓
确认结果写入 docs/项目配置.md 「第三方库决策」章节
    ↓
进入架构验证阶段
```

### 清单格式

```
| 子系统 | 方案（自研/第三方/标准库） | 候选库 | 引入理由 | 风险评估 |
|---------|--------------------------|--------|---------|---------|
| Math    | 第三方                   | GLM    | ...     | ...     |
| Logger  | 第三方                   | spdlog | ...     | ...     |
| ...     | ...                      | ...    | ...     | ...     |
```

## 9. 架构验证流程

### 触发时机

人类确认第三方库清单后，进入正式开发前。

### Step 1: PoC 骨架验证

```
开发者 Agent 执行：
1. 搭建最小 CMake 项目骨架（空模块 + 依赖关系）
2. 引入已确认的第三方库
3. 确认编译通过、模块依赖正确

验收标准：
- cmake 配置成功
- 所有模块能编译通过（即使是空实现）
- 第三方库能正确链接
```

### Step 2: 接口编译验证

```
开发者 Agent 执行：
1. 写出所有模块的核心接口头文件
2. 确认类型系统和依赖能编译通过
3. 确认跨模块引用不循环

验收标准：
- 所有接口头文件能编译通过
- 无循环依赖
- 模块间只通过接口层交互
```

### 验证完成后

```
验证通过
    ↓
协调者向人类报告验证结果（停止条件）
    ↓
人类确认架构可落地
    ↓
进入正式开发阶段
```
