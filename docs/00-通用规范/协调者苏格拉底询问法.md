# 协调者苏格拉底询问法

**版本**：v1.0

**用途**：在需求未明确时，通过由浅入深的多轮询问帮助用户深刻明确需求

---

## 核心原则

### 什么是苏格拉底式询问

苏格拉底式询问是一种**通过追问来澄清思想**的方法：
- 不直接给出答案，而是通过提问引导思考
- 从表面深入本质
- 帮助用户发现自己的真实需求
- 识别潜在的矛盾和未解决的问题

### 何时使用

| 场景 | 是否使用 |
|------|---------|
| 用户给出一个模糊的需求 | ✅ 使用 |
| 需求存在矛盾或冲突 | ✅ 使用 |
| 用户不确定想要什么 | ✅ 使用 |
| 需求已经很明确 | ❌ 不使用 |
| 用户明确拒绝回答 | ❌ 不使用 |

## 询问框架

### 层级 1：澄清问题（Clarifying）

```
目的：确保理解用户真正说的是什么

常用问题：
- "你说的[概念]具体指什么？"
- "能举个例子说明吗？"
- "你的意思是不是说...？"
- "这个需求的核心是什么？"
- "你希望达到什么效果？"
```

### 层级 2：探查假设（Probing Assumptions）

```
目的：发现用户默认的假设，这些假设可能不成立

常用问题：
- "你为什么认为[假设]？"
- "这个假设是建立在什么基础上的？"
- "如果[假设]不成立，会怎样？"
- "有没有考虑过相反的情况？"
```

### 层级 3：探查原因（Probing Reasons）

```
目的：理解用户做出判断的背后的原因

常用问题：
- "你为什么这么认为？"
- "是什么让你得出这个结论？"
- "支持这个结论的证据是什么？"
- "有没有相反的证据？"
```

### 层级 4：探查观点（Probing Perspectives）

```
目的：帮助用户从不同角度看待问题

常用问题：
- "其他人会怎么看这个问题？"
- "如果你是[另一个角色]，你会怎么想？"
- "有没有考虑过[另一个角度]？"
- "这个决定的利益相关方是谁？"
```

### 层级 5：探查后果（Probing Consequences）

```
目的：帮助用户看到决策的全部后果

常用问题：
- "如果这样做，会带来什么后果？"
- "最好的情况和最坏的情况分别是什么？"
- "这个决定会影响哪些方面？"
- "有没有你可能忽略的后果？"
```

### 层级 6：探查问题本身（Meta-Questions）

```
目的：反思问题本身是否正确

常用问题：
- "为什么这个问题对你很重要？"
- "解决这个问题能带来什么价值？"
- "有没有更好的问题需要先解决？"
- "你的最终目标是什么？"
```

## 询问流程

### 流程图

```
开始
  ↓
用户提出需求/问题
  ↓
判断需求是否明确？
  ↓
    ├─→ 是 → 继续正常流程
    │
    └─→ 否 → 启动苏格拉底询问
          ↓
        选择合适的层级问题
          ↓
        等待用户回答
          ↓
        分析回答
          ↓
        需求变清晰了？
          ↓
            ├─→ 是 → 总结确认需求 → 继续正常流程
            │
            └─→ 否 → 继续下一轮询问
                  ↓
                检查是否陷入循环？
                    ├─→ 是 → 结束询问，汇总问题
                    └─→ 否 → 继续询问
```

### 执行规则

1. **一次只问一个问题**
2. **等待用户完整回答后再继续**
3. **不要预设答案，让用户自己思考**
4. **记录用户的回答，作为需求的一部分**
5. **每轮结束后简要总结，确认理解正确**

## 询问示例

### 示例 1：模糊的需求

**用户说**："我想要一个更好的架构"

**询问过程**：

```
1. 澄清：
   "你说'更好的架构'具体指什么？是指性能更好，
   还是更容易维护，还是扩展性更强？"

2. 探查原因：
   "你为什么觉得现在的架构不够好？"

3. 探查后果：
   "如果改进架构，你希望解决什么问题？"

4. 探查观点：
   "你定义的'好'的架构是什么样的？"

5. 总结确认：
   "所以你的需求是：需要一个更容易扩展的架构，
   以支持未来的功能开发。对吗？"
```

### 示例 2：矛盾的需求

**用户说**："我既要运行速度快，又要功能多"

**询问过程**：

```
1. 澄清：
   "能具体说明'运行速度快'是指启动速度、
   响应速度还是整体性能？"

2. 探查假设：
   "你为什么认为运行速度和功能多不能同时满足？"

3. 探查后果：
   "如果必须二选一，你会选择哪个？"

4. 探查观点：
   "用户会更看重哪一点？"

5. 寻找平衡：
   "有没有可能通过其他方式同时满足？
    比如通过模块化设计，让用户选择加载哪些功能？"
```

## 协调者使用规范

### R-Soc-1：启动条件

```
满足以下任一条件时，应启动苏格拉底询问：
- 用户的需求描述模糊，存在多种理解
- 用户的需求之间存在潜在矛盾
- 用户的决策缺少明确的理由
- 用户不确定自己真正想要什么
```

### R-Soc-2：询问原则

```
1. 保持中立：不要暗示正确答案
2. 循序渐进：从简单到复杂
3. 记录过程：每次问答都记录
4. 适时总结：阶段性确认理解
5. 尊重边界：用户不愿回答则停止
```

### R-Soc-3：结束条件

```
满足以下任一条件时，可以结束询问：
- 用户需求已经明确
- 用户表示不需要继续
- 已经进行多轮（>5轮）但无进展
- 用户表现出不耐烦
```

|------|------|---------|
