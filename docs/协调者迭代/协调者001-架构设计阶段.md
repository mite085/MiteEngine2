## 历史决策

### 决策 001：第三方库引入策略

**日期**：2026年3月9日

**问题**：项目架构设计中，Core模块零外部依赖的决策存在缺陷。数学库（矩阵、向量、SIMD）和多线程（线程池、锁、原子操作）实现和维护成本高，自制库难以匹敌 Eigen、GLM、Boost.Thread 等多年打磨的库。

**决策**：对于引入开源第三方库，应当持积极态度。但需要综合评估：

| 评估维度 | 标准 |
|---------|------|
| **价值** | 降低开发时间成本 |
| **风险** | 库本身足够稳定，已被很多开源/商业项目使用 |

**具体指引**：
- 优先考虑使用成熟的开源库，而非自研
- 数学库：推荐 GLM（与GLSL语义兼容）或 Eigen（高性能）
- 线程库：可考虑 Boost.Thread 或标准库 std::thread
- 评估维度：维护成本、性能、社区支持、许可证

## 技术选型详细评估

### 1. 数学库评估

| 库 | 优点 | 缺点 | 推荐度 |
|-----|------|------|--------|
| GLM | Header-only、与GLSL兼容、社区活跃 | 编译时间较长 | ⭐⭐⭐⭐⭐ |
| Eigen | 高性能、SIMD优化好 | 学习曲线陡峭 | ⭐⭐⭐⭐ |
| 自研 | 完全控制、定制化 | 开发成本高、质量难保证 | ⭐⭐ |

**推荐**：GLM - 与GLSL语义完全兼容，减少学习成本

### 2. 线程库评估

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| std::thread (C++20) | 标准库、无外部依赖 | 功能基础 | ⭐⭐⭐⭐⭐ |
| Boost.Thread | 功能完善、跨平台 | 需要Boost依赖 | ⭐⭐⭐⭐ |
| EnTT多线程 | 与ECS集成 | 特定场景 | ⭐⭐⭐ |

**推荐**：std::thread - 充分利用C++20协程和标准库

### 3. 序列化格式评估

| 格式 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 二进制 | 高效、紧凑 | 调试困难 | ⭐⭐⭐⭐⭐ |
| JSON | 可读、易调试 | 体积大、解析慢 | ⭐⭐⭐⭐ |
| FlatBuffers | 高效、可向前兼容 | 学习曲线 | ⭐⭐⭐⭐ |
| Protobuf | 成熟、跨语言 | 需要编译步骤 | ⭐⭐⭐⭐ |

**推荐**：二进制为主（运行时）、JSON为辅（配置/调试）

### 4. 物理引擎评估

| 方案 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| SDF原生 | 精确、无需物理引擎 | 仅支持静态SDF | ⭐⭐⭐⭐⭐ |
| EnTT+自定义 | 与ECS集成 | 需要自研 | ⭐⭐⭐⭐ |
| Bullet | 成熟完整 | 集成复杂 | ⭐⭐⭐ |
| PhysX | 商业级 | 闭源、复杂 | ⭐⭐ |

**推荐**：SDF原生碰撞 + 简化物理（相位系统不需要完整物理引擎）

### 5. 资源包格式评估

| 格式 | 优点 | 缺点 | 推荐度 |
|------|------|------|--------|
| 自定义二进制 | 完全控制 | 需要开发 | ⭐⭐⭐⭐ |
| ZIP | 简单、通用 | 无压缩优化 | ⭐⭐⭐ |
| Ogre .zip | 游戏常用 | 较老 | ⭐⭐ |

**推荐**：自定义轻量包格式（类似Quake PK3）

### GitHub Actions 工作流

```yaml
# .github/workflows/build.yml
name: Build and Test

on: [push, pull_request]

jobs:
  build:
    strategy:
      matrix:
        os: [windows-latest, ubuntu-latest]
        config: [Debug, Release]

    runs-on: ${{ matrix.os }}

    steps:
      - uses: actions/checkout@v3

      - name: Configure
        run: cmake -B build -DCMAKE_BUILD_TYPE=${{ matrix.config }}

      - name: Build
        run: cmake --build build --config ${{ matrix.config }}

      - name: Test
        run: ctest -C ${{ matrix.config }} --output-on-failure

      - name: Upload Artifacts
        uses: actions/upload-artifact@v3
        with:
          name: mite-engine-${{ matrix.os }}-${{ matrix.config }}
          path: build/
```

## 架构设计总结

### 核心决策

| 决策 | 结论 |
|------|------|
| 渲染模式 | SDF Raymarching + RHI抽象 |
| 架构模式 | ECS + 分层架构 |
| 编辑器 | 独立进程 + IPC |
| 第三方库 | 积极引入成熟库 |
| 碰撞检测 | SDF原生精确碰撞 |
| 相位系统 | 位掩码 + 组件 |

### 架构优势

1. **模块化**：各模块职责清晰，可独立编译
2. **可扩展**：预留Vulkan/Metal后端接口
3. **高性能**：SDF精确碰撞、ECS数据导向
4. **开发效率**：编辑器热重载、异步加载
5. **跨平台**：Platform抽象层支持多平台

### 风险与缓解

| 风险 | 缓解措施 |
|------|----------|
| SDF性能 | LOD、相位剔除、优化算法 |
| 架构复杂度 | 接口审查、文档完善 |
| 第三方库依赖 | 评估、锁定版本 |
| 移动端性能 | 针对性优化、预留接口 |
