# 接口审查报告

**项目**：MiteEngine2 - SDF Raymarching 3D渲染引擎
**版本**：v1.0
**角色**：接口人
**日期**：2026年3月9日

---

## 1. 模块接口现状分析

### 1.1 ECS 与 渲染模块的接口

**现状描述**：
- ECS框架定义了渲染组件：`SDFGeometry`、`Material`、`Light`、`Camera`、`Renderable`
- World类提供`View()`方法用于查询组件
- RenderSystem在系统调度中优先级为900

**问题发现**：
1. 渲染系统获取ECS数据接口不明确
2. SDF几何数据生命周期管理风险
3. 数据传递效率问题

---

### 1.2 ECS 与 物理模块的接口

**现状描述**：
- 物理模块定义了`PhasePhysicsComponent`与ECS的`PhaseComponent`概念重叠
- 碰撞结果通过`CollisionOccurred`事件传递
- `RayCastResult`包含`entityId`字段

**问题发现**：
1. 组件概念重复
2. 碰撞结果回写机制缺失
3. 相位切换事件传递路径不清

---

### 1.3 资源模块 与 ECS的接口

**现状描述**：
- 资源管理模块定义`ResourceHandle<T>`模板
- ECS组件中通过Handle引用资源

**问题发现**：
1. 生命周期管理风险
2. SDF资源与SDFNode指针关系不清
3. 缺少资源就绪状态查询

---

### 1.4 Core 与 其他模块的接口

**现状描述**：
- Core模块为无依赖底层
- 其他模块直接include Core头文件使用

**问题发现**：
1. 耦合度较高
2. 内存分配策略不灵活
3. 线程池使用接口不统一

---

## 2. 发现的问题汇总

| 序号 | 接口区域 | 问题类型 | 严重程度 | 描述 |
|------|---------|---------|---------|------|
| 1 | ECS-渲染 | 接口缺失 | 高 | RenderSystem获取ECS数据接口未定义 |
| 2 | ECS-渲染 | 生命周期 | 高 | SDFNode指针与资源生命周期不同步 |
| 3 | ECS-物理 | 概念重复 | 中 | PhaseComponent与PhasePhysicsComponent重叠 |
| 4 | ECS-物理 | 接口缺失 | 中 | 碰撞结果回写ECS机制不明确 |
| 5 | 资源-ECS | 生命周期 | 高 | ResourceHandle在ECS中的引用计数管理 |
| 6 | 资源-ECS | 接口缺失 | 中 | 资源就绪状态查询接口缺失 |
| 7 | Core-其他 | 耦合度 | 中 | 缺少依赖注入机制 |

## 3. 建议的改进方案

### 3.1 ECS-渲染接口改进

定义`RenderSceneQuery`接口类：

```cpp
class IRenderSceneQuery {
public:
    virtual void QueryRenderables(uint8_t phaseMask, std::vector<RenderableData>& outResults) = 0;
    virtual Camera* GetActiveCamera() = 0;
    virtual void QueryLights(std::vector<LightData>& outLights) = 0;
};

struct RenderableData {
    Entity entity;
    const Transform* transform;
    const SDFGeometry* sdfGeometry;
    const Material* material;
};
```

### 3.2 ECS-物理接口改进

1. 统一`PhaseComponent`，物理模块使用ECS的PhaseComponent
2. 定义物理结果回写接口：

```cpp
class IPhysicsResultHandler {
public:
    virtual void OnCollisionEnter(Entity a, Entity b, const Contact& contact) = 0;
    virtual void OnCollisionExit(Entity a, Entity b) = 0;
    virtual void ApplyPhysicsResponse(Entity entity, const PhysicsResponse& response) = 0;
};
```

### 3.3 资源-ECS接口改进

定义组件资源引用基类：

```cpp
class IResourceComponent {
public:
    virtual void OnAttach(Entity e) = 0;
    virtual void OnDetach(Entity e) = 0;
};

struct Material : public IResourceComponent {
    ResourceHandle<Material> handle;
    void OnAttach(Entity e) override { handle.AddRef(); }
    void OnDetach(Entity e) override { handle.Release(); }
};
```

### 3.4 Core服务接口改进

引入服务定位器模式：

```cpp
class ServiceLocator {
public:
    template<typename T>
    static T* Get();

    template<typename T>
    static void Register(std::unique_ptr<T> service);

    static IAllocator* GetAllocator(const std::string& name = "default");
    static ThreadPool* GetThreadPool(const std::string& name = "default");
};
```

---

## 4. 总结

本次审查发现模块间接口设计存在以下主要问题：

1. **接口定义不完整**：ECS与渲染/物理模块的交互接口在架构文档中缺失或不明确

2. **生命周期管理风险**：跨模块的资源引用（特别是裸指针）存在生命周期不同步风险

3. **耦合度较高**：Core模块与其他模块的依赖关系过于紧密，缺乏抽象层

**建议**：在详细设计阶段明确定义所有跨模块接口，并补充接口契约文档。
